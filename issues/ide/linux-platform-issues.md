| 1   | **`WEBKIT_DISABLE_DMABUF_RENDERER=1` set even for Wayland**              | `src/main.rs`              | 26, 31, 35 | All three backend paths (X11, Wayland, Auto) set this env var. On Wayland, DMA-BUF renderer is the preferred path for hardware acceleration and proper VSync. Disabling it causes degraded performance (tearing, higher CPU usage). **Fix:** Only set for X11 backend path. On modern compositors (GNOME 45+, KDE 6+), DMA-BUF works correctly.                                                                                                                               |
| 2   | **`read_wayland()` bypasses Tauri Store, creates format coupling**       | `src/linux_display.rs`     | 28-34      | Reads raw JSON file via `read_to_string` while `write_wayland()` uses Tauri Store API. If Store changes serialization format, adds metadata, or uses atomic writes, `read_wayland()` breaks silently (returns `None`). **Fix:** This is inherent to the pre-app-init timing. Consider using the Store plugin's internal format or parsing its metadata.                                                                                                                       |
| 3   | **`window_created` silently swallows `with_webview` errors**             | `src/window_customizer.rs` | 21         | `let _ = webview_window.with_webview(...)` discards errors. On Linux, if GTK webview fails to initialize, error is silently dropped. **Fix:** Log the error at minimum for debugging.                                                                                                                                                                                                                                                                                         |
| 4   | **Gesture left in inconsistent state after handler destruction (Linux)** | `src/window_customizer.rs` | 28-30      | `data.as_ptr().cast()` passes raw pointer to `g_signal_handlers_destroy()`. Signal handlers are destroyed but the gesture object itself is not freed and remains attached to the widget. GTK may invoke the gesture during teardown, causing undefined behavior. Note: not a use-after-free — the gesture is not deallocated, just left in an inconsistent state. **Fix:** Use `gtk::Widget::remove_controller()` to cleanly remove gesture, or disconnect specific handlers. |
